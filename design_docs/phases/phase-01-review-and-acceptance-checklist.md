# Phase 01 Review / 验收清单

## 文件名
`design_docs/phases/phase-01-review-and-acceptance-checklist.md`

---

## 1. 本清单用途
本清单用于 `milvus-health` 项目 **Phase 01: Safety Boundary & Beta Positioning** 的：

- 设计 review
- 实现 review
- 测试 review
- 阶段验收

目标不是写大报告，而是用最少的问题，快速判断本阶段是否真的把“默认安全、RW 显式开启、cleanup 不再隐式删库”收口了。

---

## 2. 本阶段验收目标（先统一口径）
本阶段只有一个核心目标：

> 把 milvus-health 从“存在客户环境误伤风险的早期 CLI”收口为“默认安全、风险边界明确的 beta 工具”。

验收重点只看：

1. 默认是否安全
2. RW 是否显式开启
3. cleanup 是否仍有隐式删库
4. 文档 / help / 配置示例 / 实现是否一致
5. 是否有测试真正证明上述结论

---

## 3. Design Review 清单

### 3.1 产品语义
- [ ] `check` 命令的默认语义是否已唯一化为 safe check
- [ ] 是否不再存在“help/README 写 read-only，但实际默认可能写入”的歧义
- [ ] 当前工具定位是否明确为 beta / engineer-assisted tool

### 3.2 风险边界
- [ ] RW probe 是否被定义为危险模式，而不是普通默认能力
- [ ] cleanup 是否被定义为显式危险行为
- [ ] 是否明确禁止隐式 cleanup

### 3.3 冲突策略
- [ ] 若历史测试库已存在，是否有固定处理策略
- [ ] 该策略是否偏保守（fail fast / skip + warn），而不是自动猜测处理
- [ ] 是否避免“按前缀自动删库”式行为

### 3.4 阶段范围控制
- [ ] 本次设计是否只覆盖 Safety Boundary
- [ ] 是否没有把 Phase 02 的配置契约、TLS、standby 等内容混进来一起做

**Design Review 结论**
- [ ] 通过
- [ ] 需补充后再审

## 4. Implementation Review 清单

### 4.1 默认安全
- [ ] 默认执行 `check` 时不会进入 RW 路径
- [ ] 默认执行 `check` 时不会删除测试库
- [ ] 不存在通过隐式条件绕过开关进入 RW 的路径

### 4.2 RW 显式开启
- [ ] RW probe 必须通过显式开关才可执行
- [ ] 代码中 RW 入口判断集中、清晰、可读
- [ ] 不存在多个分散入口导致的漏控问题

### 4.3 cleanup 止血
- [ ] 已移除或封堵“启动即按前缀清理旧测试库”逻辑
- [ ] `cleanup=false` 时不会发生任何删除动作
- [ ] 历史测试库冲突处理符合既定策略

### 4.4 用户可见行为
- [ ] text 输出中可见 RW enabled / disabled
- [ ] text 输出中可见 cleanup enabled / disabled
- [ ] 若 json 输出也修改了，其语义与 text 不冲突

### 4.5 文档同步
- [ ] CLI help 已同步
- [ ] README 已同步
- [ ] Quickstart / 示例配置已同步
- [ ] 文档中不再把危险行为包装成 read-only

### 4.6 不越界
- [ ] 本次没有趁机做大规模重构
- [ ] 本次没有引入新的用户契约而未说明
- [ ] 本次没有顺手修一堆后续 phase 才该处理的内容

**Implementation Review 结论**
- [ ] 通过
- [ ] 需补充后再审

## 5. Test Review 清单

### 5.1 默认安全测试
- [ ] 有测试证明默认配置下不会执行 RW probe
- [ ] 有测试证明默认配置下不会 cleanup 删除资源

### 5.2 RW 开关测试
- [ ] 有测试证明显式开启 RW 才会进入 RW 路径
- [ ] 有测试证明未开启 RW 时，即使配置部分 RW 字段，也不会误进入写路径

### 5.3 cleanup 保护测试
- [ ] 有测试证明 `cleanup=false` 时绝不删除
- [ ] 有测试覆盖“历史测试库存在”场景
- [ ] 有测试证明不会隐式按前缀扫库

### 5.4 用户可见行为测试
- [ ] 有测试证明 help / usage 与实现一致
- [ ] 有测试证明 text 输出能暴露运行模式
- [ ] 若 json 输出本阶段也补了语义，有对应测试

### 5.5 测试质量判断
- [ ] 测试覆盖的是决策逻辑，而不是只覆盖表层函数
- [ ] 负向测试存在，不只是 happy path
- [ ] 没有用“人工说明”替代测试

**Test Review 结论**
- [ ] 通过
- [ ] 需补充后再审

## 6. 阶段验收清单（最终判断）
只有下面全部满足，Phase 01 才算通过：

### 6.1 行为验收
- [ ] 默认 `check` 不会真实写 Milvus
- [ ] 默认 `check` 不会删除测试库
- [ ] RW probe 必须显式开启
- [ ] cleanup 必须显式开启
- [ ] `cleanup=false` 时绝不删库

### 6.2 契约验收
- [ ] 用户不会再被“read-only”表述误导
- [ ] 文档 / help / 配置示例 / 实现一致
- [ ] 输出结果中可看出当前运行模式

### 6.3 测试验收
- [ ] 默认安全路径有测试证明
- [ ] cleanup 保护路径有测试证明
- [ ] RW 显式开启路径有测试证明
- [ ] 至少有一组负向测试证明不会误删/误写

### 6.4 范围控制验收
- [ ] 没有把本阶段拖成大改造
- [ ] 没有把 Phase 02/03 的问题混在一起做
- [ ] 剩余问题已单独记录到后续阶段，而不是在本阶段硬塞

**Phase 01 最终验收结论**
- [ ] 通过，可进入 Phase 02
- [ ] 不通过，需要返工

## 7. 建议的交叉审查分工
为了避免你再次陷入“全程红队纠察”的旧方式，本阶段建议这样分工：

### 你（项目 owner）
主要看：
- 是否真的默认安全
- 是否有客户环境误伤风险
- 文档是否会误导人

### Codex（实现者）
主要负责：
- 修改实现
- 补测试
- 给出改动摘要
- 明确哪些内容故意没做

### 第二个 agent / Gemini-cli（阶段性 challenge）
只在改完后做阶段性挑战，不全程盯：
- 是否仍存在绕过开关进入 RW 的路径
- 是否仍有隐式 cleanup
- 是否 help / README 仍与实现冲突

### 我（长期技术教练 / 总协调）
负责：
- 判断本阶段是否真收口
- 判断是否能进入下一 phase
- 防止本阶段范围失控
